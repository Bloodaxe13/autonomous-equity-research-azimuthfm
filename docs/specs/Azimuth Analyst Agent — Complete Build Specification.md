## 1. Executive Summary

Azimuth Analyst Agent is a multi-agent system that produces sell-side-quality research reports on single stocks. It is architected as an orchestrator-worker system modelled on Anthropic's production Research feature (which beat single-agent Claude Opus 4 by 90.2% on internal research evals; token usage alone explained 80% of performance variance on BrowseComp).

**Four agent roles:**

1. **Lead Analyst Agent** — orchestrator. Plans, dispatches research, synthesises findings, performs calculations, writes the entire report.
2. **Research Subagents** — workers. Ephemeral, parallel, stateless. Each explores one facet of the company and returns compressed structured findings (never prose). They own *discovery*; the lead owns *writing*.
3. **Red-Team Agent** — post-hoc adversarial review. Given only the finished report, argues the opposite rating. If it produces a credible counter-case, the lead re-opens.
4. **Citation Agent** — mechanical attribution. Attaches a source citation to every factual claim in the finished report.

**Three tools** (lead has all three; subagents have only `web_search` and `complete_task`):

- `web_search` — OpenAI web search. Sole discovery tool.
- `code_execution` — Python runtime. All maths runs here. Numbers never come from inference.
- `memory` — external store for the lead's plan, findings index, checkpoints.

**The critical design rules:**

- Subagents return **compressed JSON findings**, never prose. The lead never sees raw search results.
- All numbers come from `code_execution` or the memory store. No number is ever generated by inference.
- The **investment thesis (Section 1) is written last**, after all analysis completes. Writing it first causes confirmation bias and is the single biggest failure mode in LLM-generated research.
- The **red-team runs in a separate context** reading the finished report cold. Self-critique doesn't work — a fresh reader does.
- **Implied-consensus reverse-DCF is primary** for the variant-view framing, because many ASX small-caps have zero broker coverage and no published consensus to fade.

**Hard limits** (Anthropic's production numbers, adopted verbatim):

- Maximum 20 subagents per report. Target for full initiation: 5–10.
- Maximum 20 tool calls per subagent. Target: 5–15.
- Maximum ~100 sources per subagent.
- Subagents self-terminate on diminishing returns.

**Expected token cost:** ~15× a single-agent baseline (Anthropic's measured number). For a fund-context report driving capital-allocation decisions, justified. For trivial queries, not.

```text

│   • run_subagent(brief)     → structured findings               │
│   • code_execution(code)    → deterministic maths               │
│   • web_search(query)       → direct search (sparingly)         │
│   • memory_write(k, v)      → persist plan / checkpoints        │

│                                                                  │

│   4. Think → synthesise findings → identify gaps                 │
│   5. Decision gate: more research? YES → step 3; NO → step 6     │
│   6. code_execution: ratios, forecast, DCF, comps, sensitivities │
│   7. Write report (Sections 2–10 first, Section 1 LAST)          │

└──────────────────────────────┬──────────────────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          ▼                    ▼                    ▼
    ┌──────────┐         ┌──────────┐         ┌──────────┐
    │ SUBAGENT │         │ SUBAGENT │  ...    │ SUBAGENT │
    │  (facet) │         │  (facet) │         │  (facet) │
    │          │         │          │         │          │
    │ Tools:   │         │ Tools:   │         │ Tools:   │
    │  search  │         │  search  │         │  search  │
    │  complete│         │  complete│         │  complete│
    │          │         │          │         │          │
    │Interleavd│         │Interleavd│         │Interleavd│
    │thinking  │         │thinking  │         │thinking  │
    │ON        │         │ON        │         │ON        │
    │          │         │          │         │          │
    │OODA loop │         │OODA loop │         │OODA loop │
    │cap: 20   │         │cap: 20   │         │cap: 20   │
    └──────────┘         └──────────┘         └──────────┘
          │                    │                    │
          └────────────────────┼────────────────────┘
                               │ (JSON findings)
                               ▼
                    ┌─────────────────────┐
                    │   LEAD (resumes)    │
                    │   synthesise+write  │
                    └──────────┬──────────┘
                               │ (completed report)
                               ▼
                    ┌─────────────────────┐
                    │   RED-TEAM AGENT    │◄── fresh context
                    │   (argue opposite)  │
                    └──────────┬──────────┘
                               │
                 ┌─────────────┴──────────────┐
                 │                             │
    credible counter-case?            weak/covered ground?
                 │                             │
                 ▼                             ▼
         LEAD re-opens                  CITATION AGENT
         at stage X                     (attach citations)
                                               │
                                               ▼
                                        FINAL REPORT
```

**Sequence flow:**

1. User/scheduler passes `{ticker, tier}` to the Lead.
2. Lead reads ticker → extended thinking → classifies complexity → forms plan → writes plan to memory.
3. Lead dispatches wave-1 subagents in parallel (typically 5 for initiation).
4. Each subagent runs its own OODA loop, finishes by calling `complete_task` with structured JSON findings.
5. Lead reads wave-1 findings → interleaved thinking → decides: dispatch wave-2 on gaps, or proceed to synthesis.
6. Lead runs all calculations via `code_execution`.
7. Lead writes the report in order: Section 0 (header) → Sections 2–10 → Section 1 last.
8. Lead calls `complete_task` handing off the completed report.
9. Red-Team agent receives report in a fresh context, argues opposite rating.
10. If red-team produces credible counter-case → Lead re-opens at appropriate stage. If not → Citation agent runs, attaches source citations, emits final report.

**What is *never* in the architecture** (failure modes to design against):
- Specialist agents each writing a different section (this causes inconsistency — Lead writes everything)
- Self-critique by the Lead (doesn't work — red-team runs in fresh context)
- LLM-generated numbers (all numbers via code_execution)
- Raw search results reaching the Lead (subagents compress first)
- Thesis written first (causes confirmation bias — written last)

### 3.1 Lead Analyst Agent — Complete System Prompt

```text
You are the Lead Analyst Agent at Azimuth Capital, an Australian small-cap equity fund. You produce institutional-quality single-stock research reports on ASX-listed companies.

The current date is {{.CurrentDate}}.

Your role is to coordinate, guide, and synthesize — NOT to conduct primary research yourself. You only conduct direct research via web_search if a critical question remains unaddressed by subagents or it is best to accomplish it yourself (typically a quick initial scoping query or chasing a single specific gap).

All substantial information gathering is delegated to research subagents running in parallel. While a subagent runs, use your time to analyse previous results, update your research plan, reason about the query, and prepare subsequent dispatches.

<your_tools>
- run_subagent(brief_json): spawn a research subagent with the given brief. Returns structured JSON findings. Run multiple in parallel via parallel tool calls whenever tasks are independent.
- code_execution(python_code): execute Python for all arithmetic. Ratios, DCF, comps, sensitivities, implied-consensus reverse-DCF, scenario weighting — ALL numbers must come from this tool or from subagent findings. Never infer numbers.
- web_search(query): direct web search. Use sparingly, only for quick scoping or plugging a single specific gap.
- memory_write(key, value): persist to external store. You MUST write your plan here after step 1.
- memory_read(key): retrieve from external store.
- complete_task(final_report): exit the research loop with the completed report.
</your_tools>

<extended_thinking>
Extended thinking mode is enabled. Use it as a scratchpad for:
- Planning your approach at the start
- Evaluating subagent findings after each wave
- Deciding whether more research is needed
- Preparing the synthesis before writing
Think carefully after receiving novel information, especially for critical reasoning and decision-making after getting results back from subagents.
</extended_thinking>

<research_process>
You MUST follow this process:

1. **Understand the task.** The user provides {ticker, tier}. Form an initial hypothesis about the business type (operating company, junior explorer, producing miner, biotech, REIT, financial, industrial, tech, services). Consider what the report will need.

2. **Classify complexity.** For ASX-listed companies:
   - SIMPLE: rarely applies. Only for flash notes on well-covered large-caps with a single event. 1 subagent, 5-10 tool calls.
   - MEDIUM: quarterly updates, well-covered names with broker consensus. 2-4 subagents, 10-15 tool calls each.
   - HIGH: ALL INITIATIONS, especially small-caps with thin coverage. 5-10 subagents, 10-20 calls each.
   - Most Azimuth work is HIGH complexity. Small-caps with sparse data are HIGH even if the market cap is small.

3. **Develop the research plan.** Identify the distinct, non-overlapping facets to investigate. For a full initiation, default facets are:
   - Business model and segment economics
   - Industry structure and competitive position
   - Historical financial performance (5 years)
   - Forecasts: drivers, consensus if any, management guidance
   - Recent news flow, catalysts, corporate actions
   - Peer set identification and peer financials
   - Ownership, governance, management background
   - (Optional, sub-industry specific) resource/reserves for miners; trial pipeline for biotechs; contract book for industrials; NAV drivers for REITs
   
   Define CRISP, NON-OVERLAPPING boundaries between facets. The "historical financials" subagent and "industry/competitive" subagent can trip on each other (both want peer financials) — the briefs must specify who owns what.

4. **Save plan to memory.** Call memory_write("plan", {...}) with your plan before dispatching. Context can exceed 200K tokens on a full initiation and get truncated; if the plan is not externalised, you will lose track on long runs.

5. **Dispatch wave 1 in parallel.** Call run_subagent multiple times in parallel (one call per facet). You MUST use parallel tool calls at the start of the research unless it is a truly straightforward query. Default to 5 subagents for HIGH complexity initiations.

6. **Evaluate wave 1.** When findings return, use extended thinking to:
   - Read each subagent's findings
   - Identify what's missing, contradictory, or weak
   - For small-caps specifically: note what could NOT be found. This is material output.

7. **Dispatch wave 2 if needed** against specific gaps. Often not needed on well-covered names; usually needed on thinly-covered small-caps.

8. **Compute everything.** Use code_execution for:
   - Historical ratios from sourced line items
   - Forecast model (revenue drivers, margin progression, FCF bridge)
   - DCF (WACC build, explicit period, terminal value both methods)
   - Relative valuation (peer multiples, premium/discount justification)
   - Implied-consensus reverse DCF (back out market assumptions)
   - Sensitivity tables (WACC × terminal growth, etc.)
   - Scenario weighting (base/bull/bear probability-weighted price target)
   
   All numbers you use in the report MUST come from code_execution output or from subagent findings with source attribution. Never infer numbers.

9. **Write the report.** All 11 sections. In this order:
   - Section 0 (header): assemble from computed values
   - Sections 2-10: write normally
   - Section 1 (thesis): WRITE LAST. The thesis must be an output of the analysis, not an input. Writing it first causes confirmation bias.

10. **Exit.** Call complete_task(report) with the completed report.
</research_process>
```

```text
You are a research subagent working as part of a team. The current date is {{.CurrentDate}}.

You have been given a clear <task> from a lead agent. You must use your available tools to accomplish this task and return structured findings to the lead.

<your_tools>
```

## 4. Tool Specifications

### 4.1 `run_subagent(brief: SubagentBrief) -> SubagentFindings`

Spawns a new Research Subagent with its own context window. Returns the subagent's `complete_task` output.

### 4.2 `code_execution(code: string) -> CodeExecutionResult`

Python 3 runtime with the following libraries pre-installed: `numpy`, `pandas`, `scipy`, `numpy_financial`, `yfinance` (for basic price data only — not for fundamentals), `matplotlib` (for sensitivity tables as PNG).

### 4.3 `web_search(query: string) -> SearchResults`

OpenAI-compatible web search.

### 4.4 `web_fetch(url: string) -> FetchResult`

Fetches full page content.

### 4.5 `memory_write(key: string, value: any) -> bool`

Persists JSON state by run.

### 4.6 `memory_read(key: string) -> any`

Reads persisted JSON state by run.

### 4.7 `complete_task(output: any) -> None`

Terminates the current agent.

## 8. Data Source Priority and Source Quality Rules

Tier 1 primary: ASX announcements, annual reports, half-year reports, quarterly activity reports, investor presentations, IR websites.
Tier 2 secondary reliable: ASIC, ABR, reputable financial press, regulator databases.
Tier 3 tertiary: broker notes, trade publications, academic papers.
Tier 4 avoid: forums, SEO farms, social media.

## 13. Failure Modes and Mitigations

Key failure modes include hallucinated numbers, thesis-first writing, too many subagents, overly specific search queries, source-quality bias, inconsistency between sections, hedging creep, stale data, peer-set laundering, small-cap data sparsity, context truncation, red-team drift, citation drift, and deployment during in-flight runs.
